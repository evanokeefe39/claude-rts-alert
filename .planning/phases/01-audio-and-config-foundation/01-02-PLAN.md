<objective>
Build the settings.json merge module that can safely add Claude Code hook entries without clobbering existing user configuration.

Purpose: Claude Code stores user hooks in ~/.claude/settings.json. This module must handle four scenarios: (1) file does not exist yet, (2) file exists with no hooks, (3) file exists with other hooks that must be preserved, (4) file exists with hooks from a previous claude-notify install that should be updated. Getting this wrong destroys user configuration, so thorough testing is critical.

Output: A tested config module that reads, merges, and writes settings.json safely.
</objective>

    - readSettings(path) on nonexistent file returns empty object {}
    - readSettings(path) on valid JSON returns parsed content
    - readSettings(path) on malformed JSON throws descriptive error (not silent corruption)
    - writeSettings(path, data) creates parent directories if they do not exist
    - writeSettings(path, data) writes pretty-printed JSON (2-space indent, trailing newline)
    - mergeHooks(existingSettings, newHooks) with empty existing returns settings with only new hooks
    - mergeHooks preserves all non-hook keys in settings (e.g., "theme", "apiKey", anything)
    - mergeHooks preserves existing hooks from OTHER sources (hooks not containing "claude-notify" in command)
    - mergeHooks replaces existing claude-notify hooks (hooks containing "claude-notify" in command) with new ones
    - mergeHooks with existing hooks for same event appends new hooks rather than replacing the entire event array (only replaces claude-notify entries)
  </behavior>
  <action>
1. Create src/config/settings.ts with three exported functions:

   `readSettings(settingsPath: string): Record<string, unknown>`
   - If file does not exist, return {}
   - If file exists, read and JSON.parse
   - If JSON is malformed, throw Error with message including the path and parse error
   - Do NOT silently return {} on parse errors -- user needs to know their config is broken

   `writeSettings(settingsPath: string, data: Record<string, unknown>): void`
   - Create parent directories with fs.mkdirSync recursive
   - JSON.stringify with 2-space indent
   - Append trailing newline
   - Write atomically: write to settingsPath + '.tmp', then rename (prevents partial writes on crash)

   `mergeHooks(existing: Record<string, unknown>, newHooks: Record<string, HookEntry[]>): Record<string, unknown>`
   - Type HookEntry = { type: "command"; command: string; matcher?: string }
   - Spread all existing keys into result
   - For hooks: iterate each event key in newHooks
     - Get existing event array (or empty)
     - Filter OUT any entries whose command includes "claude-notify" (these are ours to replace)
     - Concat remaining existing entries with new entries
   - Return merged settings object

2. Create src/config/settings.test.ts using vitest:

   - Clean up temp dir in afterEach
   - Test all behaviors listed above
   - For the "preserves other hooks" test: create settings with a custom hook like { type: "command", command: "echo hello" } and verify it survives merge
   - For the "replaces claude-notify hooks" test: create settings with an existing claude-notify hook and verify it gets replaced, not duplicated

3. Create src/config/index.ts: `export { readSettings, writeSettings, mergeHooks } from './settings';`
   Also export the HookEntry type.

Follow TDD: write all tests first, verify they fail, implement, verify they pass.
