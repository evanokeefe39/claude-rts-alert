Hook handler CLI entry point: dist/hooks/handler.js
- Reads CLAUDE_HOOK_EVENT_NAME env var
- Reads CLAUDE_HOOK_TOOL_NAME, CLAUDE_HOOK_TOOL_OUTPUT for PostToolUse context
- Claude settings.json hooks format: { type: "command", command: "node /path/to/handler.js" }
- Hook commands must contain "claude-notify" for identification by mergeHooks

Settings.json hook event keys expected by Claude Code:
- "Stop" (not "stop")
- "Notification" (not "notification")
- "SessionStart" (not "session-start")
- "PostToolUse" (not "post-tool-use")

PostToolUse needs matcher: ".*" to catch all tool uses (error filtering happens in handler).

  <files>src/cli/setup.ts, src/cli/setup.test.ts, src/cli/index.ts</files>
  <behavior>
    - setup() displays a welcome message
    - setup() presents theme choices using @inquirer/select
    - Each theme choice has a name and description (e.g., "WC3 Orc Peon" with "Warcraft 3 Orc worker sounds")
    - After user selects a theme, all 4 event sounds for that theme play as preview
    - After preview, user is asked to confirm or pick again
    - On confirm: setActiveTheme is called, hooks are written to settings.json, confirmation printed
    - Hook commands use `node` + absolute path to dist/hooks/handler.js from the installed package location
    - The installed package location is derived from __dirname at runtime (path.resolve(__dirname, '../hooks/handler.js'))
  </behavior>
  <action>
    1. Install @inquirer/select as a production dependency: `npm install @inquirer/select`

    2. Create `src/cli/setup.ts`:
       - Import select from `@inquirer/select`
       - Import THEME_NAMES, setActiveTheme, getThemeSoundFile, THEMES from `../themes`
       - Import playSound from `../audio`
       - Import readSettings, writeSettings, mergeHooks, HookEntry from `../config`
       - Import path, os

       - Define THEME_DISPLAY_NAMES mapping ThemeName to friendly names:
         - 'wc3-orc' -> 'WC3 Orc Peon'
         - 'wc3-human' -> 'WC3 Human Peasant'
         - 'aoe2' -> 'Age of Empires 2'
         - 'classic-windows' -> 'Classic Windows'

       - Export async function `setup()`:
         a. Print welcome: "claude-notify setup\n" and "Choose a sound theme for Claude Code notifications:\n"
         b. Build choices array from THEME_NAMES: { name: THEME_DISPLAY_NAMES[t], value: t }
         c. Use `select({ message: 'Select a theme:', choices })` to get theme selection
         d. Print "Previewing sounds for {display name}...\n"
         e. Play all 4 event sounds sequentially with labels:
            - For each event in ['stop', 'notification', 'session-start', 'post-tool-use']:
              print "  {event}..." then play getThemeSoundFile(theme, event) via playSound
         f. Use `select({ message: 'Install this theme?', choices: [{ name: 'Yes, install', value: 'yes' }, { name: 'Pick a different theme', value: 'retry' }, { name: 'Cancel', value: 'cancel' }] })`
         g. If 'retry', loop back to step c
         h. If 'cancel', print "Setup cancelled." and return
         i. If 'yes':
            - Call setActiveTheme(theme)
            - Build hook entries: for each Claude event name (Stop, Notification, SessionStart, PostToolUse), create a HookEntry with command: `node "{handlerPath}"` where handlerPath = path.resolve(__dirname, '../hooks/handler.js'). For PostToolUse, add matcher: ".*"
            - Build newHooks object keyed by Claude event names
            - Read settings from ~/.claude/settings.json, merge with mergeHooks, write back
            - Print "Theme '{display name}' installed! Claude Code will now play sounds on events.\n"
            - Print "Run `claude-notify list` to see themes or `claude-notify uninstall` to remove.\n"

    3. Update `src/cli/index.ts`:
       - Import setup from `./setup`
       - Wire default (no args) and "setup" command to call `await setup()`

       - Test: hook commands contain "claude-notify" and "handler.js"
       - Test: PostToolUse hook entry has matcher: ".*"
       - Test: cancel flow prints cancellation message without installing
       - Use temp directories for settings.json (same DI pattern or mock readSettings/writeSettings)
