<objective>
Build the theme registry and hook event handler that connects Claude Code hook events to themed sound playback.

Purpose: This is the core wiring of Phase 2 -- when Claude Code fires a hook event (Stop, Notification, SessionStart, PostToolUse), the handler looks up the active theme, resolves the correct sound file, and plays it. The theme registry defines what sounds each theme maps to each event. Active theme preference is stored in a simple JSON config file at ~/.claude-notify/config.json (separate from Claude's settings.json to avoid coupling).

Output: Tested theme registry module and hook handler module. Sound file resolution will point to files created in Plan 02.
</objective>

1. Create src/hooks/handler.ts:

   Import playSound from '../audio', getActiveTheme and getThemeSoundFile from '../themes'.

   Map Claude Code hook event names to our HookEvent type:
   ```typescript
   const EVENT_MAP: Record<string, HookEvent> = {
     'Stop': 'stop',
     'Notification': 'notification',
     'SessionStart': 'session-start',
     'PostToolUse': 'post-tool-use',
   };
   ```

   `handleHookEvent(eventName: string, context?: { toolName?: string; toolError?: string }): Promise<void>`
   - Map eventName via EVENT_MAP. If not found, log to stderr and return (graceful no-op).
   - For PostToolUse: only play sound if context.toolError is truthy (non-empty string). If no error, return silently. This is critical -- PostToolUse fires on EVERY tool use, we only want sound on failures.
   - Get active theme via getActiveTheme().
   - Get sound file via getThemeSoundFile(theme, event).
   - Call playSound(soundFile).
   - Catch and log playSound errors to stderr (don't crash Claude Code's hook system).

   CLI entry point (runs when file is executed directly):
   ```typescript
   if (require.main === module) {
     const eventName = process.env.CLAUDE_HOOK_EVENT_NAME ?? '';
     const context = {
       toolName: process.env.CLAUDE_HOOK_TOOL_NAME,
       toolError: process.env.CLAUDE_HOOK_TOOL_OUTPUT,
     };
     // For PostToolUse, only trigger if there's an error indicator
     // Claude Code sets CLAUDE_HOOK_TOOL_OUTPUT with the tool's stderr/error output
     handleHookEvent(eventName, context).catch((err) => {
       process.stderr.write(`claude-notify: ${err}\n`);
     });
   }
   ```

   Note on PostToolUse error detection: Claude Code provides environment variables for hooks. CLAUDE_HOOK_TOOL_OUTPUT contains tool output. For error detection, we check if this contains error indicators. The `matcher` field in settings.json hooks can also filter -- but for v1, we'll use a simple approach: the PostToolUse hook command itself checks env vars. The settings.json hook entry for PostToolUse will NOT use matcher (fire on all tool uses), but the handler will check CLAUDE_HOOK_TOOL_OUTPUT for error content.

   Actually, re-reading the Claude Code hook docs pattern from settings.ts: PostToolUse hooks can have a `matcher` field. For error detection, set matcher to match error patterns. But simpler: in the handler, check if the tool execution had errors via the env vars Claude Code provides. Use process.exitCode or CLAUDE_HOOK_TOOL_OUTPUT presence as error signal.

   Simplest correct approach: The handler reads CLAUDE_HOOK_TOOL_OUTPUT. If this env var is set and non-empty, treat it as an error and play the error sound. If empty or unset, skip.
